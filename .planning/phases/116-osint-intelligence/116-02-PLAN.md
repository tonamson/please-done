# Phase 116 Plan 02: OSINT Integration with pd:audit

## Objective

Integrate OSINT libraries into the pd:audit skill with tiered command support.

## Wave 2: Integration

### Task 1: OSINT Aggregator Module
```yaml
action: |
  Create bin/lib/osint-aggregator.js that coordinates all OSINT operations:

  1. Implement OsintAggregator class:
     - constructor(options) - {cache, rateLimit, outputFormat}
     - gather(target, options) - main entry point for OSINT gathering
     - getResults() - returns aggregated findings

  2. Target parsing:
     - Support domain targets (example.com)
     - Support URL targets (https://example.com)
     - Extract domain from URL automatically
     - Validate target format

  3. Integration with Phase 116-01 libraries:
     - GoogleDorks - generate dork queries
     - CtScanner - discover subdomains
     - SecretDetector - scan for exposed secrets
     - SubdomainOsint - aggregate subdomain data

  4. Result aggregation:
     - Combine findings from all sources
     - Add timestamps and source provenance
     - Deduplicate findings across sources
     - Score risk/confidence levels

  5. Progress reporting:
     - Emit events for long-running operations
     - Support progress callbacks
     - Handle timeouts (default 5 min per source)

read_first:
  - bin/lib/google-dorks.js
  - bin/lib/ct-scanner.js
  - bin/lib/secret-detector.js
  - bin/lib/subdomain-osint.js
  - bin/lib/recon-cache.js

acceptance_criteria:
  - bin/lib/osint-aggregator.js exists and exports OsintAggregator
  - gather() returns structured report with {dorks, subdomains, secrets, metadata}
  - Integrates at least 3 OSINT sources
  - Deduplicates findings across sources
  - Respects rate limits and timeouts
  - Unit tests: test/lib/osint-aggregator.test.js
```

### Task 2: OSINT Report Generator
```yaml
action: |
  Create bin/commands/osint-report.js for CLI output:

  1. Implement report generation:
     - generateJSON(report) - structured JSON output
     - generateTable(report) - CLI table format
     - generateMarkdown(report) - markdown for documentation
     - generateSummary(report) - brief overview

  2. Report sections:
     - Executive Summary - high-level findings count
     - Google Dorks - actionable search queries
     - Subdomains Discovered - list with confidence scores
     - Secrets Found - high-risk findings with locations
     - Recommendations - next steps based on findings

  3. Risk scoring:
     - Critical: Live secrets (AWS keys, tokens)
     - High: Subdomains exposing services
     - Medium: Information disclosure via dorks
     - Low: Generic dork queries available

  4. Output options:
     - stdout (default)
     - File output (--output file.json)
     - Append to existing report (--append)

read_first:
  - bin/lib/osint-aggregator.js
  - skills/audit.md

acceptance_criteria:
  - bin/commands/osint-report.js exists with all formatters
  - JSON output is machine-parseable
  - Table output is human-readable in terminal
  - Markdown output includes all findings sections
  - Risk scores are consistent and documented
  - Can write to file or stdout
```

### Task 3: Update pd:audit Skill
```yaml
action: |
  Update skills/audit.md to include OSINT flags:

  1. Add flag documentation:
     --osint          Quick OSINT scan (Google Dorks + CT logs)
     --osint-full     Comprehensive OSINT (all sources + secret detection)
     --osint-output   Output format: json, table, markdown
     --osint-timeout  Timeout per source (default: 300s)

  2. Document command examples:
     pd:audit --osint example.com
     pd:audit --osint-full example.com --osint-output json
     pd:audit --recon --osint example.com  # Combined recon

  3. Update tiered command system:
     - FREE tier: --osint (basic dorks + CT logs)
     - STANDARD tier: --osint (same as FREE)
     - DEEP tier: --osint-full (comprehensive)
     - RED TEAM tier: --osint-full with extended timeout

  4. Document integration with recon-cache:
     - OSINT results cached for 24h
     - Cache key: osint:{domain}:{scope}
     - --fresh flag to bypass cache

  5. Add success criteria for OSINT mode:
     - Generates actionable Google Dorks
     - Discovers subdomains via CT logs
     - Identifies exposed secrets (full mode)
     - Respects rate limits

read_first:
  - skills/audit.md
  - docs/pentest/PTES.md
  - bin/lib/flag-parser.js

acceptance_criteria:
  - skills/audit.md updated with OSINT flag documentation
  - Flag behavior matches tiered system
  - Examples show usage patterns
  - Integration with recon-cache documented
  - Success criteria defined for OSINT operations
```
