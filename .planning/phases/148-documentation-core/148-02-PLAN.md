# Plan 148-02: Full rewrite of COMMAND_REFERENCE.md

## Goal
Replace the stale 34-line `docs/COMMAND_REFERENCE.md` (bulleted list, broken links, ~12 commands) with a complete reference document covering all 20 PD commands grouped into 5 categories, each with **Purpose**, **Syntax**, and **Example** blocks.

## Context
The current COMMAND_REFERENCE.md is outdated: it uses a bulleted list with `(commands/*.md)` links that don't resolve, covers only ~12 of the 20 commands, and uses old syntax like `pd init` instead of `/pd:init`.

This plan executes **after** Plan 148-01 (Wave 1) but the two files are independent — no content from cheatsheet is copied here; both use the authoritative command table from locked decisions.

Locked decisions in effect:
- **D-04**: Full rewrite of COMMAND_REFERENCE.md
- **D-05**: Format per command: `### \`pd:command\`` + **Purpose** + **Syntax** + **Example**
- **D-06**: 5 categories — Project / Planning / Execution / Debug / Utility
- **D-07**: Realistic illustrative examples (from command descriptions, not invented)
- **D-10**: New header, no external links, no `commands/` subdirectory references

## Tasks

### Task 1: Write the new COMMAND_REFERENCE.md

**File(s)**: `docs/COMMAND_REFERENCE.md`

**Action**: Overwrite `docs/COMMAND_REFERENCE.md` entirely with the content below. Copy it verbatim — do not abbreviate, omit commands, or add extra links.

```markdown
# PD Command Reference

Quick reference for all 20 PD commands. Each entry: purpose, syntax, and one example.

---

## Project Commands

Manage project lifecycle from initialization to milestone completion.

### `pd:onboard`

**Purpose**: Orient an AI agent to an unfamiliar codebase — reads structure, detects stack, and creates `.planning/` with PROJECT.md, SCAN_REPORT.md, and CONTEXT.md.

**Syntax**: `/pd:onboard [project path]`

**Example**: `/pd:onboard ./my-project`

---

### `pd:init`

**Purpose**: Initialize the planning workspace and verify FastCode MCP is available. Creates the `.planning/` directory structure and compact context files.

**Syntax**: `/pd:init [project path]`

**Example**: `/pd:init`

---

### `pd:scan`

**Purpose**: Scan the entire project, analyze structure, detect libraries and frameworks, and surface security concerns. Produces SCAN_REPORT.md.

**Syntax**: `/pd:scan [project path]`

**Example**: `/pd:scan`

---

### `pd:new-milestone`

**Purpose**: Strategic project planning — define goals, create ROADMAP.md with phases and requirements for a new milestone.

**Syntax**: `/pd:new-milestone [milestone name] [--reset-phase-numbers]`

**Example**: `/pd:new-milestone v2.0`

---

### `pd:complete-milestone`

**Purpose**: Complete the current milestone: commit all changes, create a git tag, and generate a completion report.

**Syntax**: `/pd:complete-milestone`

**Example**: `/pd:complete-milestone`

---

### `pd:sync-version`

**Purpose**: Sync the version from `package.json` across README badges, doc headers, and other version references in the repository.

**Syntax**: `/pd:sync-version [--check]`

**Example**: `/pd:sync-version --check`

---

## Planning Commands

Research, plan, and gather documentation before writing code.

### `pd:plan`

**Purpose**: Technical planning and task breakdown for the current milestone phase. Creates RESEARCH.md, PLAN.md, and TASKS.md.

**Syntax**: `/pd:plan [--auto | --discuss] [phase]`

**Example**: `/pd:plan --discuss 3.1`

---

### `pd:research`

**Purpose**: Automated research — classify questions as internal vs external, collect evidence, and produce a RESEARCH.md with findings and recommendations.

**Syntax**: `/pd:research [research topic]`

**Example**: `/pd:research "best approach for rate limiting in Express"`

---

### `pd:fetch-doc`

**Purpose**: Download documentation from a URL using the current library version. Saves docs locally for offline reference during planning.

**Syntax**: `/pd:fetch-doc <URL> [custom-name]`

**Example**: `/pd:fetch-doc https://docs.stripe.com/api stripe-api`

---

### `pd:update`

**Purpose**: Check for a new version of the PD skill set on GitHub, show the changelog, and optionally apply the update.

**Syntax**: `/pd:update [--check | --apply]`

**Example**: `/pd:update --check`

---

## Execution Commands

Write code and run tests according to the plan.

### `pd:write-code`

**Purpose**: Write code for tasks already planned in TASKS.md, then lint, build, and commit. Supports sequential, parallel, and resume modes.

**Syntax**: `/pd:write-code [task number] [--auto | --parallel | --resume]`

**Example**: `/pd:write-code 3 --auto`

---

### `pd:test`

**Purpose**: Write tests and run verification, confirm results with the user, and report failures with actionable context.

**Syntax**: `/pd:test [task number | --all | --standalone path]`

**Example**: `/pd:test --all`

---

## Debug Commands

Investigate bugs, audit code, and analyze project conventions.

### `pd:fix-bug`

**Purpose**: Find and fix bugs using the scientific method: reproduce, hypothesize, verify, fix. Creates BUG_REPORT.md with reproduction steps.

**Syntax**: `/pd:fix-bug [bug description]`

**Example**: `/pd:fix-bug "login returns 500 after password reset"`

---

### `pd:audit`

**Purpose**: View and search the discussion context audit trail — list, filter, and view stored discussion summaries by phase, keyword, or date range.

**Syntax**: `/pd:audit [--phase N] [--search keyword] [--from DATE] [--to DATE] [--view N] [--json] [--limit N]`

**Example**: `/pd:audit --search "auth" --limit 5`

---

### `pd:conventions`

**Purpose**: Analyze the project and create `CLAUDE.md` with project-specific coding conventions, style rules, and architectural patterns.

**Syntax**: `/pd:conventions`

**Example**: `/pd:conventions`

---

## Utility Commands

Check status, get suggestions, and maintain project health.

### `pd:status`

**Purpose**: Display a read-only project status dashboard: current milestone, active phase, task progress, and recent activity.

**Syntax**: `/pd:status [--auto-refresh] [--refresh-threshold=N]`

**Example**: `/pd:status --auto-refresh`

---

### `pd:what-next`

**Purpose**: Check project progress and suggest the next logical command to run based on current state.

**Syntax**: `/pd:what-next [--execute]`

**Example**: `/pd:what-next`

---

### `pd:stats`

**Purpose**: Display comprehensive project statistics: total phases, plans, requirements, milestones, and completion percentages.

**Syntax**: `/pd:stats [--json]`

**Example**: `/pd:stats`

---

### `pd:health`

**Purpose**: Diagnose planning directory issues — detect missing required files, validate STATE.md structure, and report configuration problems.

**Syntax**: `/pd:health [--json]`

**Example**: `/pd:health`

---

### `pd:discover`

**Purpose**: Discover MCP tools and built-in tools across all configured platforms, listing available capabilities for the current environment.

**Syntax**: `/pd:discover [--verbose] [--json]`

**Example**: `/pd:discover --verbose`
```

The file must end with a single trailing newline after the last line. Do not add any extra footer, links, or `commands/` directory references.
