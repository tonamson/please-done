# Requirements: Please-Done v1.3

**Defined:** 2026-03-23
**Core Value:** Every workflow step must produce the highest quality code output while consuming the minimum tokens and time — the AI must never miss critical steps and must always use the correct libraries from the user's project.

## v1.3 Requirements

Requirements for Truth-Driven Development. Each maps to roadmap phases.

### Truth Protocol

- [x] **TRUTH-01**: Bảng Truths trong plan template phải có cột "Business Value" (tại sao logic tồn tại) và "Edge Cases" (trường hợp biên)
- [x] **TRUTH-02**: Mỗi Task trong tasks template phải có mục "Logic Reference" trỏ tới mã hiệu Truth (T1, T2...)
- [x] **TRUTH-03**: CHECK-04 (Truth-Task coverage) nâng severity từ WARN lên BLOCK — AI không được tạo Task không có Truth tương ứng

### Logic-First Execution

- [x] **EXEC-01**: write-code workflow có Bước 1.7 "Re-validate Logic" — AI in ra targeted paraphrase của Business Logic trước khi sửa file (~100 tokens)
- [x] **EXEC-02**: verification-report template đổi cấu trúc từ "Test Case Passed" sang "Truths Verified" với bằng chứng (Log/Screenshot/Test) đi kèm mỗi Truth

### Knowledge Correction

- [x] **CORR-01**: fix-bug workflow có Bước 6.5 "Logic Update" — nếu bug do logic sai, sửa PLAN.md (cập nhật Truth) trước khi sửa code
- [x] **CORR-02**: progress template có mục "Logic Changes" ghi lại các thay đổi nghiệp vụ phát sinh trong quá trình làm

### Logic Audit

- [x] **AUDIT-01**: plan-checker.js có hàm checkLogicCoverage (CHECK-05) — WARN cho tasks thiếu Truths, BLOCK cho Truths thiếu tasks. Code "mồ côi" = Technical Debt

## Future Requirements

### Advanced Logic Enforcement

- **AUDIT-02**: Auto-detect orphan code changes via git diff cross-reference with Truth map
- **AUDIT-03**: Logic coverage percentage reporting per phase
- **EXEC-03**: Token budget enforcement for Re-validate Logic output

## Out of Scope

| Feature | Reason |
|---------|--------|
| LLM-as-judge for Truth quality | Plan already in context, calling another LLM is circular |
| Real-time Truth validation during plan creation | Enforcement is post-authoring via plan-checker only |
| New framework rules | Focus on workflow engine logic enforcement, not adding more rules |
| Breaking changes to skill names | Backward compatibility requirement |
