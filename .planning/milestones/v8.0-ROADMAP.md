### Phase 76: Lint Recovery & Status Dashboard

**Goal:** Developers can recover from repeated lint failures without manual intervention, and get an instant project status view with a single command.
**Depends on:** Nothing (first v8.0 phase, self-contained workflow changes)
**Requirements:** LINT-01, STATUS-01
**Plans:** 2/2 plans complete

Plans:
- [x] 76-01-PLAN.md — Lint failure tracking (write-code.md Step 5 counter + Step 1.1 recovery + progress.md template)
- [x] 76-02-PLAN.md — pd:status skill (commands/pd/status.md + workflows/status.md dashboard)

**Success Criteria:**
1. After 3 consecutive lint failures in `write-code.md`, `lint_fail_count: 3` and `last_lint_error` are saved to PROGRESS.md
2. When `lint_fail_count >= 3`, resuming `write-code.md` offers a lint-only mode that skips code re-generation and re-runs lint directly
3. After the 3rd failure the workflow surfaces a `pd:fix-bug` suggestion so the user has a clear recovery path
4. `pd:status` prints a read-only 8–12 line dashboard (milestone, phase, plan, task counts, blockers) using the Haiku model and makes zero writes to any planning file

### Phase 77: Codebase Map Staleness Detection

**Goal:** Developers are always warned before using a stale codebase map, so code generation never silently reads outdated structure data.
**Depends on:** Nothing (workflow-only change, no code dependency)
**Requirements:** STALE-01
**Plans:** 3/3 plans complete

Plans:
- [x] 77-03-PLAN.md — Wave 0: test scaffold (smoke-codebase-staleness.test.js)
- [x] 77-01-PLAN.md — META.json write step in pd-codebase-mapper.md
- [x] 77-02-PLAN.md — Step 0 staleness check in scan.md

**Success Criteria:**
1. After every `pd-codebase-mapper` run, a `META.json` file is written alongside the map containing a `mapped_at_commit` SHA field
2. `scan.md` Step 0 reads `META.json`, runs `git log --oneline <sha>..HEAD`, and emits a warning when commit-delta exceeds 20
3. The warning message names the exact commit count and prompts the user to re-run `pd:scan` before continuing
4. Projects with no prior `META.json` or no git history skip the check without error

### Phase 79: Structured Agent Error Logging

**Goal:** Every agent error produces a machine-readable JSONL record so failures can be queried, aggregated, and fed into future tooling (REPLAY-01, v9.0).
**Depends on:** Nothing (new pure module + I/O wrapper, no upstream dependency)
**Requirements:** LOG-01
**Plans:** 3/3 plans complete

Plans:
- [x] 79-03-PLAN.md — RED test stubs for log-schema and log-writer (Wave 1)
- [x] 79-01-PLAN.md — Pure-function log schema module bin/lib/log-schema.js (Wave 2)
- [x] 79-02-PLAN.md — Thin I/O wrapper bin/log-writer.js (Wave 3)

**Success Criteria:**
1. `bin/lib/log-schema.js` exports `createLogEntry()` and `validateLogEntry()` as pure functions with zero `require('fs')` calls
2. `bin/log-writer.js` appends validated JSONL entries to `.planning/logs/agent-errors.jsonl` on each error event
3. Every written entry contains all seven required fields: `timestamp`, `level`, `phase`, `step`, `agent`, `error`, `context`
4. `validateLogEntry()` rejects entries missing required fields — they are never written to disk

## Milestone Summary

**Key Decisions:**
- Haiku model for pd:status — cheaper model appropriate for read-only dashboard with no writes
- pd:status has no next-step suggestions — display-only; pd:what-next handles recommendations
- scan.md Step 0 is non-blocking — all paths continue to Step 1 via silent skip or informational warning
- pd-codebase-mapper Step 6 skips META.json silently when git unavailable; never writes null SHA
- JSDoc comments must not contain literal `require('fs')` strings — purity test uses plain string match on source
- Sonnet model for pd:onboard — complex git analysis + multi-step orchestration requires more capable model
- `appendLogEntry` accepts optional second arg for logFile — enables test isolation without polluting .planning/logs/
- Inline fixtures only in integration-contracts.test.js — zero live .planning/ reads per TESTING.md policy
- `validateLogEntry` imported from log-schema rather than hand-rolling 7-field check in contracts tests

**Issues Resolved:**
- Lint failures with no recovery path (LINT-01 delivers 3-strike + pd:fix-bug escalation)
- No quick project status view (STATUS-01 delivers pd:status read-only dashboard)
- Stale codebase maps used silently without warning (STALE-01 delivers commit-delta check)
- New project onboarding required 3+ manual commands (ONBOARD-01 consolidates into single command)
- Agent errors unstructured/unqueryable (LOG-01 delivers JSONL schema + writer infrastructure)
- No CI guard for cross-skill artifact format breakage (INTEG-01 delivers 28-test contract suite)

**Issues Deferred:**
- REPLAY-01 (`pd:replay [phase]`) — requires LOG-01 stable for ≥1 milestone first; deferred to v9.0
- `appendLogEntry` call-site wiring in agent workflows — Phase 79 delivers infrastructure only; wiring deferred
- DIFF-01 (`pd:diff-milestone`) — deferred to v9.0
- HOTREL-01 (hot-reload config.json) — deferred to v9.0
